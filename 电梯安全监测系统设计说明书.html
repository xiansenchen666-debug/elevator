1. 微控制器内核与时基 (MCU Kernel)

系统运行的确定性依赖于高精度的时基。STM32F103 的 TIM3 被配置为系统主心跳，负责所有运动学计算的时间基准。

1.1 时钟树与寄存器级配置

通过配置 RCC_CFGR 寄存器，确保 APB1 总线运行在 36MHz (由 72MHz 系统频率 2 分频得到)。TIM3 挂载在 APB1 上，但其倍频器检测到分频系数不为 1，故 TIM3 的输入频率自动恢复为 72MHz。

timer_reg_config.c
// 定时器寄存器级初始化逻辑 void TIM3_Int_Init(u16 arr, u16 psc) { RCC->APB1ENR |= 1 << 1; // 开启 TIM3 时钟 TIM3->PSC = psc; // 设置预分频器: 7199 (72MHz / 7200 = 10KHz) TIM3->ARR = arr; // 设置自动重装载值: 9 (1ms 溢出一次) TIM3->DIER |= 1 << 0; // 允许更新中断 (UIE) TIM3->CR1 |= 1 << 0; // 使能计数器 (CEN) // NVIC 中断优先级配置 (Group 2) // Preemption: 0 (最高), SubPriority: 1 NVIC_Init(...); }
设计要点： TIM3 的优先级必须高于串口中断（USART1），以防止在进行大数据量发送时导致运动学采样丢失或时标偏移。

2. 异步信号捕获链 (Interrupt Processing)

电梯楼层传感器通过外部中断（EXTI）实现微秒级响应，避免了主循环轮询带来的延迟不确定性。

电平触发与消抖 (Hardware Layer)

传感器输出下降沿进入 EXTI15_10 中断。代码内部使用 500ms 软屏蔽窗口，过滤电梯门运动带来的杂散反射信号。

速度差分计算 (Algorithm Layer)

记录当前 sys_tick_ms 与 last_floor_time。通过公式 ΔP/ΔT 实时更新 current_speed_cm_s。

数据同步与原子性 (OS Layer)

更新全局 volatile 变量。在主循环读取前，通过 __disable_irq() 开启临界区保护，防止 32 位数据读取过程中发生中断撕裂。

本系统最具工业价值的地方在于：从传感器物理触发，到上位机 UI 变红定格，全链路耗时严格控制在 5ms 以内。 以下 Mermaid 时序图展现了整个系统全链路的交互细节：

3. 深度运动学模型 (Kinematics Engine)

系统不仅仅是被动接收信号，它在两层楼之间通过物理模型“脑补”电梯的实时动态。

核心：二阶估算算法 (Second-order Estimation)

当电梯离开传感器 A 后，系统进入估算态。我们假设电梯在极短时间内（200ms）保持匀速运动，利用下式推导位移：

S_{estimated} = S_{floor\_base} \pm \int_{t_{trigger}}^{t_{now}} v(t) dt \approx S_{floor\_base} \pm \frac{v_{avg} \cdot (t_{now} - t_{trigger})}{1000}

底层 C 代码实现 (sensor.c 摘录)

sensor.c - Position Interpolation
float Get_Estimated_Position_M(void) { float base_pos = current_floor * FLOOR_HEIGHT_M; // 基础楼层绝对高度 (如 2楼 * 3.0m = 6.0m) if (Is_Floor_Blocked()) { // 如果当前正处于传感器遮挡区，则不进行插值，直接返回楼层绝对高度 return base_pos; } // 计算离开传感器后的时间积分 uint32_t time_since_trigger = sys_tick_ms - last_floor_time; float offset_m = (current_speed_cm_s * time_since_trigger) / 100000.0f; // cm/s -> m // 运动方向判断：如果是向下运行，速度可能被记录为负数或系统状态字为 RUN_DOWN if (elevator_state == STATE_RUN_DOWN) { return base_pos - offset_m; } else { return base_pos + offset_m; } }

安全边界修正 (Safety Clipping)

  • 楼层越限保护： 若估算位置超过下一层楼 10cm 但未收到感应信号，系统自动锁定估算值，触发“传感器丢失”报警。
  • 零漂消除： 电梯静止 2 秒后，强行将 current_speed 归零，防止由于传感器抖动积累的微小速度残余导致位置漂移。

4. 上位机：大规模异步处理 (Host Architecture)

基于 ElevatorMonitor.py 的上位机不再是一个简单的显示器，它是一个高性能的数据处理中枢。采用 CustomTkinter 现代 GUI 库，完全摒弃了传统 Tkinter 的老旧组件，构建了“国际范浅色高级风”的控制面板。

4.2 生产者-消费者多线程模型与线程安全

为了保证 UI 在高频串口通信下不卡顿（无假死），系统彻底分离了 IO 密集型任务和 GUI 渲染任务。

线程类型 核心任务与实现机制 跨线程同步方案
串口守护线程
Daemon Thread		 通过 threading.Thread(target=self.read_serial_data, daemon=True) 启动。使用 readline().decode('utf-8') 阻塞读取，剥离非法字符。 严禁直接操作 UI！必须打包数据并下发给主线程。
GUI 主线程
Mainloop		 解析 JSON 字典，更新 CTkProgressBar、CTkLabel 的颜色与数值。计算 pos / SHAFT_HEIGHT_M 进行进度条归一化。 使用 self.after(0, self.update_dashboard, data) 将任务安全推入 Tkinter 事件队列。
架构亮点： 为什么不使用 queue.Queue？因为 Tkinter 内置的 after(0, func, *args) 底层已经实现了线程安全的事件循环投递，这是 Python GUI 跨线程更新的最优雅解法，避免了锁竞争带来的性能损耗。

4.4 边缘触发的“报警定格”算法

在实际工业现场，超速往往发生在一瞬间（可能仅持续几十毫秒）。上位机代码中实现了一个精妙的边缘触发器 (Edge Trigger)，用于在 UI 上永久定格那一瞬间的“罪魁祸首”。

# 冗余判断：无论数据包中的 alarm、relay 还是 state，只要有一个报警即视为报警 alarm_active = bool(data.get("alarm", 0)) or state == "ALARM" # 边缘检测：只有当 上一时刻安全(False) 且 当前时刻报警(True) 时触发 if alarm_active and not self.alarm_active: speed_val = data.get("speed", 0.0) self.trigger_speed = speed_val # 内存定格 self.log(f"安全保护触发：检测到超速 ({speed_val} m/s)") # UI 定格：显示红色的警告标签，直到人工点击 RESET self.trigger_speed_label.configure(text=f"⚠️ 触发超速记录: {speed_val} m/s") self.trigger_speed_label.grid() # 呼出隐藏控件 self.alarm_active = alarm_active # 更新状态机

4.5 异步 UI 渲染管线 (Asynchronous UI Pipeline)

在传统的阻塞式 GUI 编程中，如果在主循环中执行 serial.read()，会导致整个界面冻结。本系统采用了一种被称为“单向数据流与事件投递”的异步管线架构。

事件循环投递机制 (Event Loop Injection)

Python 的 queue.Queue 虽然线程安全，但需要 UI 线程不断去轮询（Polling），这极大地消耗了 CPU 资源。本系统利用 Tkinter 底层的 Tcl/Tk 事件循环，通过 self.after(0, callback, args) 将更新任务直接“塞入”UI 线程的待办队列中。

ElevatorMonitor.py - Async Pipeline
# [线程 1: 串口守护线程 Daemon] - 绝对禁止操作任何 UI 控件 def read_serial_data(self): while self.serial.is_open: line = self.serial.readline().decode('utf-8').strip() data = json.loads(line) # 解析为字典 # 核心跨线程投递：将解析好的数据包 data 作为参数，投递给主线程的 update_dashboard 函数 # 这里的 0 代表 0 毫秒延迟，即“立刻排队” self.after(0, self.update_dashboard, data) # [线程 2: GUI 主线程 Mainloop] - 只负责纯粹的渲染 def update_dashboard(self, data): # 此时该函数已经运行在安全的主线程中 self.card_speed.configure(text=f"{data['speed']} m/s") self.progress_bar.set(data['pos'] / SHAFT_HEIGHT_M)

5. 故障模式分析与保护 (FMEA & Reliability)

系统针对工业现场可能出现的异常情况建立了详尽的响应矩阵。

底层硬件看门狗 (IWDG)

STM32 开启独立看门狗，时钟源自内部低速 LSI (40KHz)。 若主循环因 JSON 格式化错误陷入死循环，看门狗将在 2.5 秒后强行复位 MCU， 并将继电器置于初始安全状态。

UI 异常隔离 (Try-Except Sandbox)

在 update_dashboard 方法中，整个 JSON 解析与控件更新被包裹在全局的 try...except 沙盒中。即使单片机因串口干扰发送了类似 {"floo":3, "spe 的断帧残缺 JSON，json.loads 会在子线程捕获异常，或者主线程缺少键值时安全抛弃，绝不会导致上位机闪退崩溃。

5.2 报警锁定逻辑 (Alarm Latching)

安全报警具有“记忆效应”。一旦触发超速，继电器将保持断开，直到上位机发送合法的 RESET 指令。这种设计符合工业安全规范中的“手动确认复位”原则。

单片机端：硬件死锁 (main.c)

if (current_speed > ALARM_SPEED_THRESHOLD) { alarm_triggered = 1; Relay_Set(ON); // 物理切断安全回路 } // 只要没有收到上位机 RESET 指令，永远保持断开 if (alarm_triggered) { Relay_Set(ON); }

上位机端：软解围 (ElevatorMonitor.py)

def send_reset_cmd(self): # 发送 JSON 指令解除硬件死锁 cmd = json.dumps({"cmd": "RESET_ALARM"}) self.serial.write((cmd + '\n').encode()) # 解除 UI 软锁定 self.alarm_active = False self.trigger_speed_label.grid_remove() self.log("系统已人工复位，继电器重新闭合")

5.3 多重冗余保护机制 (Redundancy Protection)

在生命安全关键（Safety-Critical）系统中，单一节点的失效绝不能导致整个系统的崩溃。本系统在软件和硬件层面上实现了“三位一体”的冗余防护。

物理层：常闭继电器

安全回路继电器采用常闭（NC）触点设计。这意味着如果单片机彻底烧毁、电源断电或线路断裂，继电器将瞬间释放（断开安全回路），电梯立即物理抱死。这遵循了“故障导向安全 (Fail-Safe)”的最高原则。

逻辑层：状态机或(OR)判断

上位机在判断是否报警时，不单纯依赖某一个字段。alarm_active = bool(data.get("alarm")) or state == "ALARM"。即使下位机 JSON 封包时某个字段出错，只要有任何一个维度表明处于危险状态，系统立即强制锁定报警。

通信层：断帧免疫

电梯井道存在强烈的电磁干扰。上位机 JSON 解析置于严格的 try...except 沙盒中。若收到被干扰打断的残缺帧 {"spee": 1.2，上位机会直接静默丢弃该帧。由于控制周期高达 1ms，丢弃少数几帧完全不影响安全性。
